# Issue #51 M3.R1a — Verified Try/Catch + DLQ emission (concrete plan)

## Summary

Un-gate the zero-retry DLQ path in the existing `ProcessFlowBuilder` (which lives at `src/boomi_mcp/categories/components/builders/process_flow_builder.py`, **not** the `tools/process_builders/...` path the architect plan names — that path does not exist in this repo). When `reliability.retry_count == 0` and `reliability.dlq.mode` is `document_cache_ref` or `error_subprocess_ref`, the builder wraps the normal Start→source→[transform]→target→Stop flow in a verified `catcherrors` Try/Catch shape (`catchAll="true" retryCount="0"`, Try dragpoint `identifier="default"`, Catch dragpoint `identifier="error"`) whose Catch path emits a `doccacheload` (DLQ) shape into a second Stop. `PROCESS_RETRY_UNVERIFIED` keeps firing for any `retry_count > 0` (M3.R1b out of scope). The issue #28 DLQ "fragments" are **config dicts**, not raw XML — `DlqWriterPrimitive.emit_fragment` already produces exactly the `reliability.dlq` shape the builder reads, so "consuming the fragment" means reading `reliability.dlq` (and optional `reliability.error_classifier`) keys the builder already receives; no primitive-XML plumbing is needed. Finally promote `reliability.on_failure` out of `deferred_fields` in the process schema template (`src/boomi_mcp/categories/meta_tools.py`), leaving `execution.trigger`/`execution.run_metadata` deferred.

## File-by-file

### 1. `src/boomi_mcp/categories/components/builders/process_flow_builder.py` (core change)

**Reliability gate — `_validate_reliability(reliability)` (lines 637–711).**
- Keep the `retry_count > 0` branch (lines 663–674) firing `PROCESS_RETRY_UNVERIFIED` unchanged (M3.R1b is BLOCKED).
- In the `dlq_mode != "disabled"` branch (lines 698–710), **stop raising `PROCESS_RETRY_UNVERIFIED`** for the two supported modes. New behavior:
  - If `dlq_mode in ("document_cache_ref", "error_subprocess_ref")` **and** `retry_count == 0` → return `None` (allowed). Note `retry_count` is already validated/known at this point because the `retry_count > 0` guard returns earlier, so reaching this branch implies `retry_count == 0`.
  - Add a binding check so the catch path has something to emit: for `document_cache_ref`, require exactly one of `dlq.document_cache_id` / `dlq.document_cache_ref_key` (non-empty string); for `error_subprocess_ref`, require exactly one of `dlq.process_id` / `dlq.process_ref_key`. On violation return a `BuilderValidationError(error_code="PROCESS_DLQ_BINDING_INVALID", field="reliability.dlq.<...>")`. This mirrors `DlqWriterParameters._check_mode` (operational.py lines 412–447) so the builder rejects the same malformed shapes the primitive already rejects, keeping the two layers consistent.
  - Leave `disabled` returning `None` (unchanged); leave unknown modes returning `PROCESS_DLQ_BINDING_INVALID` (unchanged, lines 690–697); leave non-dict `dlq` returning `PROCESS_DLQ_BINDING_INVALID` (unchanged).
- Add a classmethod helper `ProcessFlowBuilder._should_emit_try_catch(reliability)` returning `True` when `reliability` is a dict, `retry_count == 0`, `dlq` is a dict, and `dlq.mode in {"document_cache_ref","error_subprocess_ref"}`. Reused by `build()`.

**`$ref` reachability in `validate_config` (lines 273–311).** The existing `_walk_scalars`/depends_on loop already covers `reliability.dlq.document_cache_ref_key` / `process_ref_key` **only if** those values are written as `$ref:KEY` tokens. The `DlqWriterPrimitive` writes a bare key (e.g. `"document_cache_ref_key": "my_cache"`), not a `$ref:` token, and appends it to `depends_on`. Decision: the builder does **not** need to re-validate the ref-key against depends_on (integration_builder owns dependency ordering and id substitution). Keep the existing `$ref` loop untouched. The DLQ binding id used in emitted XML is whatever the config carries (a real component id after apply-time `$ref` substitution, or a literal id); the builder emits it verbatim like every other id, stripped of whitespace.

**`build()` — wrap the flow (lines 319–499).**
- Compute `emit_try_catch = cls._should_emit_try_catch(config.get("reliability"))` after building the existing `flow` list (lines 360–414) but before the shape-emission walk.
- When `emit_try_catch` is `False`: the existing code path is used **verbatim** so all existing golden/ElementTree tests in `test_process_flow_builder.py` stay byte-identical (acceptance: "Existing non-DLQ XML unchanged").
- When `emit_try_catch` is `True`, restructure shape emission:
  - The Start shape's outgoing dragpoint must point at the new `catcherrors` shape instead of the first connectoraction.
  - Insert a `catcherrors` shape between Start and the source connectoraction. It carries **two** dragpoints (built via a new `_emit_trycatch_dragpoints` helper, see below): the Try dragpoint with `identifier="default"` → first normal shape (the source connectoraction), and the Catch dragpoint with `identifier="error"` → first catch-path shape.
  - The normal chain (source → [transform] → target → Stop) is unchanged in structure but its shape indices shift by one (Start=shape1, catcherrors=shape2, source=shape3, …) and the normal-chain terminal Stop is `continue="true"`.
  - The Catch path: emit a `doccacheload` shape (DLQ) whose single dragpoint points at a **second** Stop shape. For `document_cache_ref` the `doccacheload` carries the document-cache id; for `error_subprocess_ref` emit the error-subprocess invocation shape instead (see emitter note). Optionally, when `reliability.error_classifier` is present, emit a classifier shape (e.g. a `decision`/`message` per the captured native pattern) before the DLQ shape — but keep R1a minimal: emit the classifier shape only if the captured `dff0bf83-...` reference actually has one in the catch leg; otherwise wire `error` → `doccacheload` → Stop directly. Resolve this by inspecting the live reference component during implementation (read-only `manage_component get` on `dff0bf83-d525-4571-b572-c93d285bb788` under the `work` profile, per CLAUDE.md "ground plans in live Boomi MCP data").
  - Because shape names are positional (`shape{N}`), assign deterministic indices: Start=1, catcherrors=2, normal source=3, …, normal Stop=K, then catch shapes continue K+1 (`doccacheload`/classifier) and K+2 (catch Stop). Keep the `_shape_x`/`_dragpoint_x` geometry helpers; give the catch shapes a distinct `y` (e.g. `_SHAPE_Y + 96`) so the diagram is readable (geometry is decorative per the file's own comment, lines 50–58).
  - Keep the closing `ET.fromstring(component_xml)` round-trip guard (lines 489–497) — it validates the new shapes too.

**New shape emitters (add alongside `_emit_start_noaction` etc., lines 718–839):**

Confirm the exact `image` attribute and any extra `catcherrors` config attributes against the live `dff0bf83-...` export before finalizing.

The existing `_emit_dragpoints` (lines 819–839) does not emit `identifier`, so a dedicated helper is needed for the Try/Catch shape; do **not** change `_emit_dragpoints` (keeps every existing dragpoint test passing).

**Module-level: nothing new exported.** `PROCESS_FLOW_BUILDERS`, `get_process_flow_builder`, and `__all__` (lines 881–909) are unchanged. The builder remains the single consumer; no new registry entries.

## Deviations from architect plan

- **Paths corrected.** This repo has no `tools/process_builders/` tree. The real builder is `src/boomi_mcp/categories/components/builders/process_flow_builder.py`; schema templates live in `src/boomi_mcp/categories/meta_tools.py` (not `tools/build_integration.py`). All edits are mapped to the real files.
- **"Primitive fragment XML" placeholder resolved.** The issue #28 `dlq_writer`/`error_classifier` primitives (`src/boomi_mcp/patterns/primitives/operational.py`) emit **config dicts** via `emit_fragment`, not raw fragment XML or components (`emit_components` returns `[]`). `DlqWriterPrimitive`'s output is exactly the `reliability.dlq` shape the builder already reads, so the builder consumes the fragment by reading `reliability.dlq` config keys it already receives — no primitive import, no `primitives/__init__.py` export, no compat shim. This avoids a `builders → patterns` import cycle.
- **Golden fixture style.** The repo deliberately commits no byte-exact process XML fixtures and tests structure via ElementTree. Recommending ElementTree structural assertions (or, if a committed golden is required by the issue, a normalized-XML compare) instead of byte-for-byte, to match repo convention and avoid attribute-order brittleness. Reason: consistency with the existing `test_process_flow_builder.py` and lower maintenance risk.
- **Existing gate test must change.** `test_rejects_dlq_document_cache_mode` (current `test_process_flow_builder.py:443`) asserts the gate this issue removes; it is updated to assert acceptance, not merely added alongside.
